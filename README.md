# migrate

ActiveRecord-style schema migrations for Go — small, idiomatic, and built
entirely on the standard library (`database/sql`). No third-party dependencies,
no cgo.

```go
import "github.com/malcolmston/migrate"
```

## Features

- **Versioned, reversible migrations** identified by a `uint64` version, applied
  in ascending order and rolled back in descending order.
- **Go or SQL directions** — a migration's up/down can be a Go function
  `func(ctx, *sql.Tx) error` or raw SQL text (multiple `;`-separated statements).
- **Transaction per migration** — each migration runs in its own transaction. A
  failure rolls that transaction back, does not record the version, and halts
  the run. Re-running `Migrate` is idempotent.
- **Full command set**: `Migrate`, `Up(n)`, `Down(n)`, `Rollback(steps)`,
  `MigrateTo(version)`, `Redo`, and `Status`.
- **Directory / `io/fs` loading** of `<version>_<name>.up.sql` /
  `.down.sql` files, plus programmatic `Register` for Go migrations.
- **ActiveRecord-flavoured schema DSL** that emits portable ANSI SQL:
  `CreateTable`, `AddColumn`, `AddIndex`, `DropTable`, `RenameColumn`, and more.

## Quick start

```go
db, _ := sql.Open("sqlite", "app.db") // any database/sql driver

mg := migrate.New(db)
mg.Register(
    migrate.Migration{
        Version: 20240101,
        Name:    "create_users",
        UpSQL: migrate.CreateTable("users", func(t *migrate.Table) {
            t.String("email", migrate.NotNull(), migrate.Unique())
            t.Timestamps()
        }),
        DownSQL: migrate.DropTable("users"),
    },
    migrate.Migration{
        Version: 20240102,
        Name:    "add_users_name",
        Up: func(ctx context.Context, tx *sql.Tx) error {
            _, err := tx.ExecContext(ctx,
                migrate.AddColumn("users", "name", "VARCHAR", migrate.Limit(100)))
            return err
        },
        Down: func(ctx context.Context, tx *sql.Tx) error {
            _, err := tx.ExecContext(ctx, migrate.DropColumn("users", "name"))
            return err
        },
    },
)

if err := mg.Migrate(context.Background()); err != nil {
    log.Fatal(err)
}
```

### Loading migrations from files

Files are named `<version>_<name>.up.sql` and `<version>_<name>.down.sql`
(the down file is optional):

```
migrations/
  20240101_create_users.up.sql
  20240101_create_users.down.sql
  20240102_add_users_name.up.sql
  20240102_add_users_name.down.sql
```

```go
migs, err := migrate.LoadDir("migrations")
if err != nil { log.Fatal(err) }
mg := migrate.New(db)
mg.Register(migs...)
mg.Migrate(ctx)
```

`LoadFS` accepts any `io/fs.FS`, so an embedded filesystem works too:

```go
//go:embed migrations/*.sql
var migrationsFS embed.FS
sub, _ := fs.Sub(migrationsFS, "migrations")
migs, _ := migrate.LoadFS(sub)
```

### Commands

| Method                    | Effect                                                        |
| ------------------------- | ------------------------------------------------------------- |
| `Migrate(ctx)`            | Apply all pending migrations, ascending.                      |
| `Up(ctx, n)`              | Apply the next `n` pending (`n <= 0` = all).                  |
| `Down(ctx, n)`            | Roll back the `n` most recent applied, descending.            |
| `Rollback(ctx, steps)`    | Convenience alias for `Down`; `steps <= 0` rolls back one.    |
| `MigrateTo(ctx, version)` | Move to exactly `version` (up or down). `0` rolls back all.   |
| `Redo(ctx)`               | Roll back the latest applied migration and re-apply it.       |
| `Status(ctx)`             | Report every migration and whether it has been applied.       |

### Schema DSL

```go
migrate.CreateTable("posts", func(t *migrate.Table) {
    t.String("title", migrate.NotNull())
    t.Text("body")
    t.Boolean("published", migrate.NotNull(), migrate.Default(false))
    t.References("author", migrate.WithForeignKey(), migrate.ReferenceNotNull())
    t.Timestamps()
})
```

produces:

```sql
CREATE TABLE posts (
	id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
	title VARCHAR(255) NOT NULL,
	body TEXT,
	published BOOLEAN NOT NULL DEFAULT FALSE,
	author_id BIGINT NOT NULL,
	created_at TIMESTAMP NOT NULL,
	updated_at TIMESTAMP NOT NULL,
	FOREIGN KEY (author_id) REFERENCES authors (id)
)
```

Other helpers: `AddColumn`, `DropColumn`, `RenameColumn`, `RenameTable`,
`DropTable`, `DropTableIfExists`, `AddIndex`, `DropIndex`.

## SQL dialect assumptions

The generated SQL and the migrator's own bookkeeping target a conservative ANSI
subset. Choices are deliberately explicit and documented so the output is
predictable:

- **Bind parameters** use the `?` placeholder style (MySQL / SQLite / the
  in-memory test driver). Drivers requiring `$1` positional parameters (such as
  `lib/pq`) need an adapter.
- **Auto-increment primary keys** are emitted as
  `BIGINT GENERATED BY DEFAULT AS IDENTITY` (SQL:2003 identity columns). Suppress
  with `WithoutID()`.
- **Column types** map to `VARCHAR(n)`, `TEXT`, `INTEGER`, `BIGINT`, `BOOLEAN`,
  `TIMESTAMP`, `DATE`, `DOUBLE PRECISION`.
- **References** follow the Rails convention: a `<singular>_id` column and naive
  `+s` pluralisation of the referenced table (override with `ReferenceTable`).
- The bookkeeping table (default `schema_migrations`) is
  `(version BIGINT NOT NULL PRIMARY KEY, applied_at TIMESTAMP NOT NULL)`.

## The in-memory test driver

The test suite ships a real but minimal `database/sql/driver` registered as
`memdb` (see `memdb_test.go`). It implements the actual
`driver.Driver`/`Conn`/`Stmt`/`Tx`/`Rows`/`Result` interfaces so the migrator
exercises genuine `database/sql` code paths — connection pooling, transactions,
prepared statements, and row scanning. It understands the migrator's bookkeeping
SQL (create table, insert/delete/select versions), stages transactional
mutations so a rollback truly discards them, and records every committed
mutating statement in an ordered log the tests assert against. It is compiled
only for tests and is not part of the importable library.
